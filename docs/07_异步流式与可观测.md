# 第 7 章：异步流式与可观测

> 目标：理解流式输出、Token 统计、执行追踪——让 Agent 的运行"看得见"。

## 7.1 这一章解决什么问题

前面几章的 Agent 都有个共同点：**黑盒**。你调用 `think()`，等几秒，拿到一个字符串。中间发生了什么？调了几次 LLM？每次花了多少 token？哪一步最慢？一无所知。

本章解决三个"看不见"的问题：
1. **等得太久** → 流式输出，边生成边看
2. **不知道花了多少钱** → Token 统计
3. **复杂协作像黑盒** → 执行追踪（Trace）

## 7.2 流式输出（chat_stream）

### 问题
默认 `chat()` 是阻塞的——必须等 LLM 把整个答复生成完才返回。长答复要干等好几秒，体验差。

### 解决：chat_stream + stream_callback

`GLMClient` 提供流式接口，逐块产出文本：

```python
def chat_stream(self, messages, tools=None, tool_choice="auto"):
    """流式调用，逐块 yield StreamChunk"""
    response = self.client.chat.completions.create(**kwargs, stream=True)
    for chunk in response:
        yield StreamChunk(delta_content=chunk.delta.content, ...)
```

Agent 通过 `stream_callback` 暴露流式能力：

```python
def on_chunk(text):
    print(text, end="", flush=True)   # 每收到一段就实时打印

agent.think("介绍量子纠缠", stream_callback=on_chunk)
# 输出会像打字一样逐字显现
```

### 机制细节
- 流式**只用于最终答复**。ReAct 循环中的工具调用步骤仍用非流式（因为要解析 tool_calls 结构）。
- 当模型决定给出最终答复时（不再调工具），`think` 用 `chat_stream` 重新请求，逐块回调。
- 流式失败时自动回退到非流式，保证可用性。

## 7.3 Token 统计（TokenUsage）

### 问题
Hierarchical 团队一次任务可能调用 5-10 次 LLM，烧了多少 token 毫无感知，无法核算成本。

### 解决：自动累计

每次 LLM 调用的 usage（prompt/completion/total tokens）会被提取并累计：

```python
agent.think("复杂问题")
print(agent.get_token_usage())
# TokenUsage(prompt_tokens=850, completion_tokens=320, total_tokens=1170)

agent.think("另一个问题")
print(agent.get_token_usage().total_tokens)   # 累加后的总数
```

### 机制
- `ChatResult.usage` 字段携带本次调用的用量
- `Agent._token_usage` 在每次 think 的每轮循环中累加（`.add()`）
- 流式时从最后一个 chunk 的 usage 累计
- `reset_token_usage()` 可重置

## 7.4 执行追踪（Tracer）⭐

### 问题
Team 跑一个任务，内部 leader 规划、成员执行、汇总……到底经过了哪些步骤？每步多久？哪步最贵？没有追踪，复杂协作完全是黑盒。

### Tracer 设计

`Tracer`（`core/trace.py`）是一个**线程局部**的追踪器，通过 enable/disable 开关控制：

```python
class Tracer:
    _enabled = False                  # 全局开关
    _local = threading.local()        # 线程局部（每线程独立 trace）
```

**零开销设计**：默认关闭。关闭时所有记录方法（start_span/end_span 等）第一行就 `if not cls._enabled: return None`，不分配内存、不计时、不构造对象。生产环境无任何性能损失。

### Trace 与 TraceSpan

```python
@dataclass
class TraceSpan:
    name: str            # 如 "think:助手" 或 "hierarchical:researcher"
    start: float         # 开始时间戳
    end: float           # 结束时间戳
    attributes: dict     # 自由属性（steps, total_tokens, member 等）
    parent_id: str       # 父 span（构建嵌套）
```

```python
@dataclass
class Trace:
    spans: List[TraceSpan]    # 所有 span
    def format_text(self)     # 格式化为时间线文本
    def total_tokens(self)    # 汇总所有 span 的 token
    def to_dict(self)         # JSON 序列化
```

### 用法

```python
from core import Tracer

Tracer.enable()
Tracer.start_trace("任务")
agent.think("复杂问题")
# 或 team.run("团队任务")
trace = Tracer.end_trace()

print(trace.format_text())
```

输出示例（团队任务）：
```
=== Trace: 任务 ===
总耗时: 31.12s   总 token: 3987   span 数: 10

  think:项目经理          6.35s  total_tokens=225     (规划)
  hierarchical:researcher 2.86s  capability=search    (子任务1)
  think:研究员            2.86s  total_tokens=357
  hierarchical:writer     2.40s  capability=write     (子任务2)
  think:作家              2.40s  total_tokens=298
  ...
  think:项目经理          7.29s  total_tokens=736     (汇总)
```

每一行都是一个可观测的步骤——谁、做了什么、多久、多少 token。黑盒变透明。

### 插桩点
- `Agent.think`：每次推理开一个 span（记录 steps、total_tokens）
- `SequentialProcess` / `HierarchicalProcess`：每个成员的处理开一个 span
- `Team.run`：整个任务开一个 trace（自动，无需手动）

### Span 嵌套
start_span 维护一个线程局部的栈，新 span 的 parent_id 设为栈顶。end_span 弹栈。这构建出 parent/child 树，能表达"think 内部的工具调用"这种嵌套关系。

## 7.5 AsyncRouter 的可观测性

AsyncRouter（第 5 章介绍）除了根治栈溢出，也保留了和同步 Router 一样的统计：

```python
router.get_router_stats()
# {"agent_count": 3, "stats": {"messages_sent": 199, "messages_received": 199, "routing_errors": 0}}
```

异步路由的每条消息处理仍被计入统计，不因异步而失去可观测性。

## 7.6 三个能力如何配合

一个生产级的 Agent 应用，三者通常一起用：

```python
Tracer.enable()                    # 开追踪
Tracer.start_trace("用户请求")
chunks = []
result = agent.think(
    user_input,
    stream_callback=lambda s: chunks.append(s),   # 流式
)
trace = Tracer.end_trace()
print(f"耗时 {trace.duration:.1f}s, token {trace.total_tokens()}")  # 统计
```

## 7.7 与其他模块的关系

- 流式基于 `GLMClient.chat_stream`（第 2 章用到）
- Token 由 `ChatResult.usage` 提取，在 `Agent.think` 累计（第 2 章）
- Tracer 插桩在 `Agent.think` 和 `Process.execute`（第 6 章）
