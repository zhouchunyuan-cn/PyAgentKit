# 第 6 章：团队协作

> 目标：理解多个 Agent 如何组成"团队"，按流程协作完成复杂任务。

## 6.1 这一章解决什么问题

第 5 章的 Router 让 Agent 间能互发消息，但那是"消息路由"——谁发给谁靠 Agent 自己在代码里 `send`。如果要换一种协作流程，就得改 Agent 的代码。

**Team（团队）** 是更高层的抽象：把"成员有哪些"和"按什么流程协作"分开。成员就是普通 Agent，流程由 **Process** 决定。换个 Process 就换种协作方式，无需改 Agent。

## 6.2 Team 与 multi-agent 的区别

| | multi-agent（Router） | Team（Process） |
|---|---|---|
| 关注点 | 谁发给谁（消息路由） | 按什么顺序、谁做什么（流程编排） |
| 数据传递 | 各自独立的 memory | **SharedContext 共享黑板** |
| 流程 | 写死在 Agent 的 send 里 | 由 Process 声明，可切换 |
| 适用 | 松散的、对等的 Agent 通信 | 有共同目标、有分工的团队 |

## 6.3 核心组件

### SharedContext——团队共享黑板

团队成员通过它共享中间产物，而非各自独立的 memory：

```python
class SharedContext:
    task: str                          # 团队共同目标
    outputs: Dict[agent_id, str]       # 各成员的产出
    intermediate: Dict[str, Any]       # 中间产物（如 Leader 的规划）
    history: List[Dict]                # 执行步骤记录
```

关键方法 `get_all_outputs()`：拼接所有成员的产出，作为下一个成员的输入上下文。

### Process——流程策略

抽象基类，定义"团队成员怎么协作"：

```python
class Process(ABC):
    def execute(self, task, members, leader=None) -> str:
        """执行团队任务，返回最终结果"""
```

两种内置实现：SequentialProcess（顺序接力）和 HierarchicalProcess（Leader 编排）。

### Team——团队

```python
class Team:
    def __init__(self, name, members, process, leader=None)
    def run(self, task_description) -> str   # 执行任务
```

## 6.4 SequentialProcess——顺序接力

最直观的流程：成员按声明顺序执行，前者的产出作为后者的上下文。

```
任务："写一篇科普文章"
  │
  ├─ 成员1（研究员）: 收到任务 → 思考 → 产出"研究要点"
  │     ↓ 产出存入 SharedContext
  ├─ 成员2（作家）: 收到"任务 + 研究员的要点" → 思考 → 产出"文章初稿"
  │     ↓ 产出存入 SharedContext
  └─ 成员3（审核员）: 收到"任务 + 前两人产出" → 思考 → 产出"终稿"
        ↓
     最终结果 = 成员3的产出
```

每个成员的输入由 `_build_input` 组装：
- 第一个成员：只收到原始任务
- 后续成员：收到"团队任务 + 此前所有人的累计产出"

```python
from core import Team, SequentialProcess

team = Team(
    name="内容组",
    members=[researcher, writer, reviewer],
    process=SequentialProcess(),
)
result = team.run("写一篇关于量子计算的科普文章")
```

**适用**：固定的确定性流水线（研究→写作→审核）。

## 6.5 HierarchicalProcess——Leader 自动编排 ⭐

这是 Team 模块的精华。Leader 用 LLM **动态分析任务、自动拆解、按能力分配成员、最后汇总**。

### 三阶段流程

```
任务："完成一份 AI 行业分析"
  │
  ├─ 阶段1: 规划（Leader）
  │    Leader 收到任务 + 成员能力清单，输出 JSON 子任务规划：
  │    [{"subtask": "调研 AI 行业数据", "capability": "search"},
  │     {"subtask": "分析市场趋势", "capability": "analysis"},
  │     {"subtask": "撰写报告", "capability": "write"}]
  │
  ├─ 阶段2: 执行（按能力匹配成员）
  │    子任务1 → 找具备 search 能力的成员 → think → 产出存入 context
  │    子任务2 → 找具备 analysis 能力的成员 → think（带着子任务1的产出）→ 产出
  │    子任务3 → 找具备 write 能力的成员 → think（带着全部产出）→ 产出
  │
  └─ 阶段3: 汇总（Leader）
       Leader 收到"任务 + 所有成员产出"，think 出最终结果
```

### Leader 的规划 prompt

Leader 被要求输出严格的 JSON 数组：
```
你是项目经理。任务：{task}
成员及能力：{members}
请拆解为子任务，每个指定所需能力。只输出 JSON：
[{"subtask": "...", "capability": "..."}]
```

`_parse_plan` 提取 JSON、校验字段、过滤无效项。解析失败时回退为"每个成员处理一个通用子任务"，保证健壮性。

### 能力匹配

`_find_member_by_capability(members, capability)` 找到第一个声明了该能力的成员。这正是第 2 章讲的 `capabilities` 的用武之地——**协作系统按能力选人，而非靠名字硬编码**。

```python
from core import Team, HierarchicalProcess

team = Team(
    name="项目组",
    members=[researcher, analyst, writer],
    process=HierarchicalProcess(max_subtasks=4),
    leader=manager,   # manager 声明了 capabilities=["plan"]
)
result = team.run("完成一份 AI 行业分析")
```

## 6.6 两种流程怎么选

| | Sequential | Hierarchical |
|---|---|---|
| 流程 | 固定顺序 | Leader 动态决定 |
| 确定性 | 高（每次一样） | 低（Leader 可能拆出不同子任务） |
| 智能 | 低（机械接力） | 高（理解任务后规划） |
| 成本 | 低（N 次 think） | 高（N + 规划 + 汇总次 think） |
| 适用 | 已知流水线 | 不确定该谁做什么的复杂任务 |

## 6.7 用 YAML 配置团队（无需写代码）

```yaml
# configs/team_example.yaml
llm:
  model: glm-4-flash
agents:
  - id: leader
    capabilities: [plan]
  - id: researcher
    capabilities: [search]
    tools: [web_search]
  - id: writer
    capabilities: [write]
team:
  name: 内容组
  process: hierarchical
  leader: leader
  members: [researcher, writer]
```

```bash
pyagentkit run configs/team_example.yaml
# 进入团队交互模式，输入任务即触发 Leader 编排
```

详见第 8 章。

## 6.8 与其他模块的关系

- 成员就是普通 Agent，复用 `think()`（第 2 章）
- 能力匹配依赖 `Agent.capabilities`（第 2 章）
- `SharedContext` 替代了 Router 场景下各自独立的 memory（第 5 章）
- 执行过程可被 Tracer 追踪（第 7 章）
- YAML 配置由 ConfigLoader 解析（第 8 章）
