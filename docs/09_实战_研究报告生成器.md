# 实战：构建智能研究报告生成器

> 本篇是一个**端到端的项目实战**。我们将从零构建一个完整的应用：输入一个主题，
> 团队自动调研、分析、撰写、审核，产出一份结构化研究报告。
>
> 通过这个项目，你会把前 8 章学到的能力全部串起来用一遍：
> **ReAct 推理 + 工具调用 + 记忆 + 团队协作 + 执行追踪**。

---

## 项目目标

做一个**智能研究报告生成器**：

```
输入："人工智能在医疗领域的应用"
                │
                ▼
   ┌─────────────────────────────┐
   │   团队（Leader 自动编排）      │
   │                             │
   │  Leader: 拆解任务            │
   │    ├─ 研究员: 联网搜索资料     │
   │    ├─ 分析师: 解读关键趋势     │
   │    └─ 作家: 撰写结构化报告     │
   │  Leader: 汇总终稿            │
   └─────────────────────────────┘
                │
                ▼
输出：一份带标题、要点、结论的研究报告
```

## 你将学到

- 如何设计多 Agent 团队的角色分工
- 如何用 capabilities 驱动 Leader 自动分配
- 如何用 web_search 工具让 Agent 联网获取真实信息
- 如何用 Tracer 观察整个协作过程（谁做了什么、耗时多少）
- 如何用 YAML 配置让一切可复现、可调整

---

## 第 1 步：理解需求与角色设计

一份好的研究报告需要四个角色协作：

| 角色 | 能力 (capabilities) | 职责 | 工具 |
|------|---------------------|------|------|
| **项目经理** | `plan` | 拆解任务、分配、汇总 | 无（纯推理） |
| **研究员** | `search` | 联网搜集资料 | `web_search` |
| **分析师** | `analysis` | 解读资料、提炼趋势 | 无 |
| **作家** | `write` | 整理成结构化报告 | 无 |

为什么这样分？因为 Hierarchical 流程的 Leader 会按 `capability` 匹配成员——我们让每个角色声明不同的能力，Leader 就能把"搜索"子任务分给研究员、"分析"分给分析师，各司其职。

---

## 第 2 步：用 YAML 声明团队（声明式）

创建配置文件 `configs/report_generator.yaml`：

```yaml
# 智能研究报告生成器配置
llm:
  model: glm-4-flash

agents:
  # 项目经理：负责拆解任务和汇总
  - id: leader
    name: 项目经理
    system_prompt: |
      你是一个经验丰富的项目经理。
      收到研究任务后，你会把它拆解为若干子任务，每个子任务指定所需的能力。
      最后你会汇总所有成员的产出，形成连贯的最终报告。
    capabilities: [plan]

  # 研究员：联网搜集资料
  - id: researcher
    name: 研究员
    system_prompt: |
      你是专业研究员。收到调研任务后，使用 web_search 工具搜索真实信息，
      然后整理成要点。务必基于搜索到的真实内容，不要编造。
    capabilities: [search]
    tools: [web_search]

  # 分析师：解读资料
  - id: analyst
    name: 分析师
    system_prompt: |
      你是数据与行业分析师。基于研究员提供的资料，
      提炼关键趋势、机会与挑战，给出有洞察力的分析。
    capabilities: [analysis]

  # 作家：撰写报告
  - id: writer
    name: 报告撰写员
    system_prompt: |
      你是专业报告撰写员。把团队的调研和分析整理成结构清晰的报告，
      包含：标题、背景、核心发现（分点）、结论与展望。
    capabilities: [write]

# 团队：Leader 自动编排
team:
  name: 研究报告组
  process: hierarchical
  leader: leader
  members: [researcher, analyst, writer]
  max_subtasks: 4

session:
  agent: leader
```

> **关键点**：
> - `process: hierarchical` 让 Leader 用 LLM 智能分配，而非机械接力
> - 研究员挂了 `web_search` 工具，其他角色纯推理
> - `max_subtasks: 4` 限制 Leader 最多拆 4 个子任务，控制成本

## 第 4 步：用代码精细控制（编程式）

CLI 简单但不够灵活。下面用 Python 代码实现同样的功能，并加上 **执行追踪** 观察全过程。

```python
# report_generator.py
import os
from dotenv import load_dotenv
load_dotenv()

from core import GLMClient, Tracer
from core.config import ConfigLoader
from core.logging_config import setup_logging

def main():
    setup_logging()

    # 1. 加载配置并构建团队
    loader = ConfigLoader()
    config = loader.load_file("configs/report_generator.yaml")
    llm = GLMClient(model=config.model)
    _, agents, _ = loader.build(config, llm_client=llm)
    team = loader.build_team(config, agents)

    print(f"团队就绪：{team.name}，成员 {[m.agent_id for m in team.members]}\n")

    # 2. 开启执行追踪
    Tracer.enable()

    # 3. 执行研究任务
    topic = "人工智能在医疗领域的应用现状与前景"
    print(f"研究主题：{topic}\n{'='*60}\n")

    Tracer.start_trace("研究报告任务")
    report = team.run(topic)
    trace = Tracer.end_trace()

    # 4. 输出报告
    print("\n" + "="*60)
    print("最终报告：")
    print("="*60)
    print(report)

    # 5. 输出执行追踪（看团队内部发生了什么）
    print("\n" + "="*60)
    print("执行追踪：")
    print("="*60)
    print(trace.format_text())

if __name__ == "__main__":
    main()
```

---

## 第 5 步：理解执行追踪

运行上面的代码，`trace.format_text()` 会输出类似这样的时间线：

```
=== Trace: 研究报告任务 ===
总耗时: 28.5s   总 token: 3240   span 数: 9

  think:项目经理          6.1s  total_tokens=210     (拆解任务)
  hierarchical:researcher 3.2s  capability=search    (联网搜索)
  think:研究员            3.2s  total_tokens=380
  hierarchical:analyst    2.8s  capability=analysis  (分析趋势)
  think:分析师            2.8s  total_tokens=295
  hierarchical:writer     4.5s  capability=write     (撰写报告)
  think:作家              4.5s  total_tokens=540
  think:项目经理          7.1s  total_tokens=815     (汇总终稿)
```

从这个追踪你能清楚看到：
- **Leader 第一步花 6.1s 拆解任务**（输出 JSON 子任务规划）
- **研究员联网搜索**用了 web_search 工具（所以耗时长些）
- **作家写报告花 4.5s**（产出最长，token 最多）
- **Leader 最后汇总**耗时最长（要把所有产出整合成连贯报告）
- **总成本 3240 token**

这就是 Trace 的价值——把黑盒的团队协作变成**完全可观测**的过程。

## 第 6 步：调整与优化

### 优化 1：控制成本

如果觉得 token 用量太大，可以：
- 减少 `max_subtasks`（让 Leader 少拆几个子任务）
- 换用 Sequential 流程（固定流水线，比 Hierarchical 少一次"规划"调用）

```yaml
team:
  process: sequential        # 改为顺序接力
  members: [researcher, analyst, writer]
  # 不需要 leader
```

### 优化 2：让报告更结构化

调整作家的 system_prompt，要求严格的输出格式：

```yaml
- id: writer
  system_prompt: |
    你是报告撰写员。输出必须严格遵循格式：
    # {报告标题}
    ## 背景与概述
    ...
    ## 核心发现
    1. ...
    2. ...
    ## 结论与展望
    ...
```

### 优化 3：加入向量记忆（积累经验）

让团队记住历史报告，下次类似主题能复用：

```python
from core import VectorMemory, LocalTfidfEmbedder

# 给 Leader 注入向量记忆
vm = VectorMemory(embedder=LocalTfidfEmbedder())
vm.add("上次的量子计算报告：重点关注了量子纠错和退相干问题")
agents["leader"].set_vector_memory(vm)

# 现在 Leader 在规划时会"想起"相关的历史经验
```

## 第 7 步：扩展思路

这个报告生成器可以继续演进：

| 扩展方向 | 怎么做 |
|----------|--------|
| **支持文件输入** | 给研究员加 `file_read` 工具，让它读本地资料 |
| **数据支撑** | 给分析师加 `database` 工具，查询真实数据 |
| **多轮迭代** | 用 Sequential 流程加一个"审核员"角色，反复打磨 |
| **导出格式** | 报告产出后用 Python 写入 markdown/html 文件 |
| **并发加速** | 用 AsyncRouter 让多个子任务并行（需改 Team 流程） |
